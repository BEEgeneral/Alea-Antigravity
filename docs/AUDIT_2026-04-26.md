# Audit Completo — Alea Signature
**Fecha:** 26 Abril 2026  
**Auditor:** Hermes Agent  
**Alcance:** Código, arquitectura, DB, IA, tokens

---

## RESUMEN EJECUTIVO

El sistema tiene una base sólida pero adolece de **duplicación de capas IA**, **ineficiencia en tokens**, y **problemas de arquitectura de datos** que causan fricción operativa. No hay API routes propias — todo pasa por InsForge directamente desde el cliente. Hay 3 integraciones MiniMax distintas y paralelas.

---

## HALLAZGOS POR ÁREA

### 1. TOKEN IA — CRÍTICO

| # | Problema | Archivo | Impacto |
|---|----------|---------|--------|
| T1 | **3 clientes MiniMax distintos** cargando en paralelo al frontend | `ai-minimax.ts` (Anthropic) + `minimax.ts` (OpenAI) + `minimax-hermes.ts` (OpenAI+Hermes) | 3 SDKs, 3 conexiones, 3 system prompts distintos |
| T2 | **System prompt de Hermes tiene ~700 tokens hardcoded** y se reconstruye en cada chat | `minimax-hermes.ts:59-79` | ~700 tokens × cada mensaje = despilfarro |
| T3 | **No hay truncamiento de historial** — `messageHistory` crece sin límite | `minimax-hermes.ts:37` | Contexto se inflará hasta 200k tokens, costos explosivos |
| T4 | **Textos de documentos sin límite** — `analyzeImagesAndText` sube hasta 5 imágenes + 3000 chars de texto por request | `minimax.ts:157-169` | hasta 5MB de imágenes base64 por llamada |
| T5 | **`@ts-ignore` en `reasoning_split`** — el parámetro estrella de M2.7 no está tipado, puede fallar silenciosamente | `minimax-hermes.ts:133-135,252-253,296` | Reasoning puede no activarse |

**Recomendación prioritaria:** Unificar a UN solo cliente MiniMax (OpenAI SDK), con un sistema de prompts centralizado y truncamiento de historial a 50 mensajes máx.

### 2. ARQUITECTURA FRONTEND — ALTO

| # | Problema | Archivo | Impacto |
|---|----------|---------|--------|
| A1 | **No hay API routes propias** — el frontend llama directamente a InsForge desde el cliente (RLS solo) | `insforge-client.ts` como Proxy + hooks | Sin validación, sin caché, sin control de errores centralizado |
| A2 | **9 llamadas en `Promise.all` sin paginación ni límites** en `fetchAll` | `useAdminData.ts:24-34` | Carga inicial megaconsumo —trae TODO sin límite |
| A3 | **`useAdminActions` duplica `fetchLeads/fetchInvestors/fetchProperties`** de `useAdminData` — lógica scattereada | `useAdminActions.ts:18-31,96-109,170-183` | Mantenimiento frágil, inconsistencia de estado |
| A4 | **Proxy `_token` en módulo** — singleton en memoria server-side, shared mutable state entre requests | `insforge-client.ts:6-7` | Bug en Next.js 16 con React Server Components |
| A5 | **No hay caché ni SWR** — cada componente hace sus propias llamadas | Todo el frontend | Refresco constante, más peticiones, más tokens |

**Recomendación:** Crear API routes propias (`/api/leads`, `/api/investors`, etc.) que hagan de gateway — validación, caché Redis/ISR, paginación, rate limiting.

## PLAN DE MEJORA PRIORIZADO

### FASE 1 — TOKENS (此优先级最高)
> **Ahorro estimado: 40-60% reducción en costos IA**

- [ ] **T1, T2, T3, T5** → Crear `/src/lib/alea-ai.ts` unificado:
  - Un solo cliente OpenAI (baseURL api.minimax.io)
  - System prompt centralizado en constant
  - Historial truncado a 50 mensajes
  - Typing correcto para `reasoning_split`
  - Streaming con SSE
- [ ] **T4** → Limitar `analyzeImagesAndText` a 2 imágenes máx, 1500 chars texto
- [ ] **Deprecar** `ai-minimax.ts` y `minimax-hermes.ts` — dejar solo `minimax.ts` refactorizado

### FASE 2 — API GATEWAY (Ahorro red: 50%)
> **Respuesta API 3-5x más rápida por caché**

- [ ] Crear API routes: `GET/POST /api/leads`, `/api/investors`, `/api/properties`
- [ ] Cada route: paginación (límite 50), caché ISR 60s, validación Zod
- [ ] Reemplazar llamadas directas a InsForge con llamadas a localhost
- [ ] Eliminar `insforge-client.ts` Proxy — usar Supabase client directo en server

### FASE 3 — DATABASE (Ahorro: 30-70% queries)
> **Responde en <100ms en vez de 500ms+**

- [ ] **B1** → Especificar campos exactos en cada `.select()` — nunca `*`
- [ ] **B2** → Añadir `.limit(20)` a todos los JOIN con relaciones
- [ ] **B3** → Solicitar a InsForge: índice en `investors(piedra_personalidad)`, `properties(is_off_market)`, `investor_interests(investor_id)`
- [ ] **B4** → Migrar búsqueda MemPalace a `to_tsvector` o FTS si InsForge lo soporta
- [ ] **B5** → Upsert atómico para wing+room+drawer en una sola transacción
- [ ] **B6** → Usar `Promise.all` para las 3 queries de perfil

### FASE 4 — FRONTEND (Dev experience + performance)
> **TTI reducido 40%, menos re-renders**

- [ ] **A3** → Unificar `useAdminData` y `useAdminActions` en un solo hook `useCRM`
- [ ] **A4** → Eliminar Proxy singleton — crear cliente por request con `new`
- [ ] Implementar React Query / SWR para caché y dedup de peticiones
- [ ] **U4** → Mover `imap` y `mailparser` a `/src/lib/server/` (solo server components)

### FASE 5 — AGENTS
> **Agents funcionales + monitoring de tokens**

- [ ] **P1** → Fix `` `/api/agents/${agentId}/spawn` `` (cambiar a backticks)
- [ ] Implementar skills reales como herramientas ejecutables (no markdown)
- [ ] Añadir logging de tokens por agente en InsForge
